# 핵심 키워드 정리
- **역할(Role)과 구현(Implementation)**: 객체를 설계할 때 인터페이스(역할)와 클래스(구현)를 명확히 분리하는 객체 지향의 핵심 개념.
- **클라이언트(Client)와 서버(Server)**: 요청을 보내는 객체와 그 요청을 처리하여 응답하는 객체 간의 협력 관계를 의미.
- **다형성의 본질**: 실행 시점에 구현 객체를 유연하게 변경할 수 있어, 클라이언트 코드를 수정하지 않고도 서버의 기능을 확장할 수 있는 능력
- **OCP(Open-Closed Principle)**: "확장에는 열려 있고, 변경에는 닫혀 있어야 한다"는 좋은 객체 지향 설계의 5가지 원칙(SOLID) 중 하나.
- **전략 패턴(Strategy Pattern)**: 특정 알고리즘(전략)을 인터페이스로 추상화하여, 클라이언트 조건에 따라 구현체를 자유롭게 교체하는 디자인 패턴

# Java
## 1. 역할과 구현을 분리해야 하는 이유
- **세상의 단순화**: 클라이언트는 대상의 내부 구조를 몰라도 '역할'만 알면 서비스를 이용할 수 있습니다(예: 운전자는 자동차의 내부 엔진 구조를 몰라도 운전할 수 있음).
- **유연성과 변경 용이성**: 구현 대상의 내부 구조가 변경되거나, 대상 자체를 다른 것으로 교체해도 클라이언트에게 영향을 주지 않습니다.
- **확장 가능성**: 새로운 기능을 추가할 때 기존 시스템의 핵심 로직을 건드리지 않고 새로운 '구현체'만 만들어서 끼워 넣으면 됩니다.
## 2. 다형성의 본질과 객체 간의 협력
- 혼자 존재하는 객체는 없으며, 수많은 객체는 서로 **메시지**를 주고받으며 협력합니다.
- 다형성을 사용하면 **오버라이딩(Overriding)** 된 메서드가 실행 시점에 우선권을 가지므로, 인터페이스를 참조하는 클라이언트가 실제 어떤 객체인지 몰라도 각 객체의 고유 기능을 호출할 수 있습니다.
## 3. OCP(개방-폐쇄) 원칙의 실제 적용
- **Open for extension**: `Car` 인터페이스를 상속받아 `K3`, `Model3`, `NewCar` 등 새로운 기능을 무한히 확장할 수 있습니다.
- **Closed for modification**: 새로운 차가 추가되어도 이를 운전하는 `Driver` 클래스의 코드는 전혀 수정할 필요가 없습니다.
- **주의점**: 인터페이스 자체가 변하면 클라이언트와 서버 모두에 큰 변경이 발생하므로, 초기 설계 시 **인터페이스를 안정적으로 설계**하는 것이 가장 중요합니다.
## 4. 스프링(Spring)과 다형성
- 스프링의 핵심 기능인 **제어의 역전(IoC)** 과 **의존관계 주입(DI)** 은 결국 다형성을 극대화하여 활용할 수 있도록 돕는 기술입니다.
- 이를 통해 마치 레고 블록을 조립하거나 공연의 배우를 교체하듯, 구현체를 편리하고 안전하게 변경하며 대규모 소프트웨어를 개발할 수 있습니다.
